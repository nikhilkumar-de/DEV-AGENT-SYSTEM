
ENGINEERING ARCHITECTURE DOCUMENT

Development Agent System
Commit Checker  |  PR Reviewer  |  SAST Scanner


1. Introduction
This Engineering Architecture Document (EAD) describes the complete technical design of the Development Agent System. It covers system components, data flow, module design, technology choices, deployment strategy, and integration patterns.
This document is intended for developers, tech leads, and reviewers who will build, review, or maintain this system.


2. System Overview
The system is a Python application that connects to GitHub and responds to two types of triggers: a scheduled cron job (for daily commit checks) and GitHub Webhook events (for PR and push events). Based on the trigger, one or more of the three modules activates and performs its function.

The diagram below shows how all parts of the system connect. The GitHub Repository is the central source of truth. Events from it drive all three modules through either a scheduler or a webhook server.

5. Module-by-Module Design
5.1  Module 1 — Daily Commit Checker


How It Works — Step by Step
Scheduler triggers commit_checker.py at 9 PM daily.
It calls the GitHub API to get all commits from today.
It extracts the GitHub usernames of everyone who committed.
It compares that list against the configured TEAM_MEMBERS list.
Anyone not in the committed list is flagged as 'missing'.
A Slack message is sent for each missing person.
Result is appended to logs/commit_log.json.

5.2  Module 2 — PR Review Agent


How It Works — Step by Step
Developer opens a PR on GitHub.
GitHub sends a webhook POST request to our Flask server at /webhook.
webhook_server.py receives it and calls pr_reviewer.py.
The agent fetches all changed files from the PR.
For each Python file, it sends the code diff to the AI (GPT-4 / Claude).
The AI returns a structured review listing all issues found.
The agent posts the review as a comment on the PR via GitHub API.

5.3  Module 3 — SAST Scanner & Test Generator


What is SAST?
SAST stands for Static Application Security Testing. It means scanning source code WITHOUT running it to find security vulnerabilities. Think of it like an X-ray machine for code — it can see dangerous patterns that a normal human reviewer might miss.


How It Works — Step by Step
Developer pushes code or opens a PR.
GitHub webhook fires to webhook_server.py.
sast_scanner.py is called with the PR number.
Changed files are downloaded temporarily to /tmp/.
Semgrep is executed on each file using subprocess.
Semgrep returns JSON output with all findings.
The scanner parses the findings — line number, severity, description.
test_generator.py creates a Python test file for each finding.
A formatted report + test cases are posted as a PR comment.

8.3  Semgrep (Module 3)
Semgrep is executed as a subprocess command and returns JSON:

$ semgrep --config auto --json /tmp/scan_target.py

Returns JSON like:
{
  'results': [
    {
      'check_id': 'python.lang.security.sql-injection',
      'path': '/tmp/scan_target.py',
      'start': { 'line': 3 },
      'extra': {
        'message': 'Detected SQL injection via string concatenation',
        'severity': 'ERROR'
      }
    }
  ]
}

Risk
Mitigation
API keys exposed in code
Store all keys as environment variables, never in code
Webhook receives fake requests
Validate GitHub webhook signature using HMAC-SHA256
GitHub API rate limiting
Cache responses, use exponential backoff on 429 errors
Semgrep false positives
Add .semgrepignore file and allow inline suppression comments
OpenAI API cost overrun
Limit to changed files only, set max_tokens cap per request
Server exposed to internet
Use HTTPS with SSL cert (Let's Encrypt), restrict to GitHub IPs
